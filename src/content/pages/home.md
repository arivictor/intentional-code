Most developers stumble into architecture — one day you're writing a script, the next you're responsible for a system with users and consequences. The decisions compound: where does this logic belong, how do these parts talk to each other, why is this so hard to change?

This is a focused reference to the patterns and principles that help you answer those questions, implemented in idiomatic Go. Each page stands alone. Jump to what you need; come back when you need it again.
