*Most developers stumble into architecture* — one day you're writing a script, the next you're responsible for a system with users and consequences. The decisions start to compound: where does this logic belong, how do these parts talk to each other, why is this so hard to change?

This is a focused reference to the patterns and principles that help you answer those questions, implemented in idiomatic Go. If you're not sure where to start, the [Design Philosophy](/go/philosophy) section explains the thinking behind the patterns — read it first and the rest of the catalog clicks into place. If you already know the pain but not the pattern name, use the tags to find what applies.
