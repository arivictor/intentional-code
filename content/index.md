A field guide to writing Go that's easier to read, test, and change — built on one habit: **always answer why before how.** This site teaches intentional code. It is about choosing architecture and patterns on purpose, based on the problem you need to solve.

Architecture is the shape of decisions. Good architecture makes change easier. Bad architecture makes change expensive.

Architectural choices should be made with clear intent: solve a real problem that exists now. Do not optimise for a problem that might happen someday. Do not choose a pattern based only on personal preference.

If you want the deeper reasoning behind this mindset, read the introduction to [Philosophy](/philosophy).
